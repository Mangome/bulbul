## Context

当前 Bulbul 画布采用水平分页布局：每个分组占一页（viewportWidth 宽），组内 2 列瀑布流纵向排列图片，分组间通过左右键切换导航。这种大图展示模式在组内图片较多时（如 10+ 张连拍），需要大量纵向滚动才能看完整组，辨别效率低。

现有技术栈：
- Canvas 2D 渲染引擎，dirty flag + rAF 按需渲染
- 坐标系统：offsetX（水平分页偏移）+ offsetY（纵向滚动）+ actualZoom（含缩放补偿）
- 虚拟化：水平分组过滤 + 组内 Y 轴二分查找裁剪
- ImageBitmap LRU 缓存（50 条目，200MB）
- GroupTitle 代码存在但未在 renderFrame 中调用

## Goals / Non-Goals

**Goals:**
- 高密度缩略图网格：每行 5-6 张缩略图，一屏看完整组图片
- 悬浮放大镜：鼠标悬停缩略图时弹出大图预览，快速确认细节
- 纵向滚动：所有分组纵向排列，滚轮上下浏览
- 简化架构：移除水平分页、缩放补偿、分组切换动画等复杂机制
- 保留现有交互功能：选中、检测框、键盘快捷键

**Non-Goals:**
- 不实现图片间拖拽重排
- 不实现自定义网格密度调节（使用固定高密度参数）
- 不在放大镜中实现图片缩放/平移（仅静态预览）
- 不改变后端逻辑和数据结构

## Decisions

### D1: 放大镜采用 HTML overlay 而非 Canvas 2D 绘制

**选择**: HTML overlay（React 组件 + absolute 定位）

**替代方案**:
- Canvas 2D 绘制放大镜：与画布渲染循环一致，但信息排版困难、受缩放影响、占用 ImageBitmap 缓存

**理由**:
- CSS 排版天然适合文字信息展示（文件名、EXIF、评分），无需手动计算文字位置
- 不受画布缩放影响，放大镜始终以屏幕像素级精度显示
- 不占用 ImageBitmap LRU 缓存（用独立 `<img>` 元素加载 medium 质量）
- 鼠标穿透问题可通过 CSS `pointer-events: none` 解决

### D2: 纵向行式网格而非瀑布流

**选择**: 行式网格（每行等高，图片等宽排列，行高等于该行最高图的缩放高度）

**替代方案**:
- 瀑布流：每列独立高度，空间利用更紧凑但缩略图不对齐，浏览效率低
- 固定行高网格：所有行等高，最简单但横向裁剪图片

**理由**:
- 缩略图对齐便于视觉扫描和比较，是图片筛选场景的核心需求
- 等宽缩放保持完整构图，不做裁剪
- 行高等于该行最高图的高度，不浪费纵向空间

### D3: 坐标系统简化为纯纵向滚动

**选择**: 仅 `scrollY` + `zoomLevel`，offsetX 恒为 0

**替代方案**:
- 保留双向平移：类似 Figma 画布，可自由拖拽

**理由**:
- 纵向流布局天然适配单向滚动，水平方向通过布局居中自动处理
- 移除 offsetX 消除了水平分页切换动画、缩放补偿等复杂逻辑
- 单向滚动更符合照片浏览的直觉（类似文件管理器的缩略图视图）

### D4: 缩略图默认 thumbnail 质量，放大镜独立加载 medium

**选择**: Canvas 2D 绘制用 thumbnail，放大镜用 HTML `<img>` 独立请求 medium URL

**替代方案**:
- 缩略图也用 medium 质量：更清晰但内存占用 4-6 倍
- 放大镜复用 LRU 缓存中的 ImageBitmap：需要 drawImage 到临时 canvas 再转 dataURL，复杂且性能差

**理由**:
- 160px 缩略图下 thumbnail 质量足够清晰
- HTML `<img>` 直接请求后端 URL，浏览器自身管理解码和内存，不影响 Canvas LRU 缓存
- 放大镜显示/隐藏时浏览器自然释放 `<img>` 资源

### D5: 最后一行居左排列

**选择**: 分组最后一行缩略图居左排列，不居中散开

**替代方案**:
- 最后一行居中：视觉对称但与上方行不对齐
- 最后一行拉伸填满：图片变形

**理由**:
- 居左排列与上方行左对齐一致，视觉连贯
- 实现简单，无需额外计算

## Risks / Trade-offs

- [放大镜延迟] 首次悬停时需加载 medium 图片，可能有短暂空白 → 用缩略图做占位，加载完成后平滑替换
- [缩略图清晰度] 160px thumbnail 在大屏幕上可能略模糊 → 通过放大镜弥补细节确认需求
- [布局重写范围大] layout.ts + viewport.ts + InfiniteCanvas.tsx 同时重写 → 按顺序实施，每步验证
- [纵向滚动分组定位] 移除水平分页后，当前分组索引需要通过 scrollY 计算 → 使用 layout pages 的 offsetY 做二分查找
